# EARS

# 1. Autenticación

## 1.1 Solicitud de acceso

**EARS-AUTH-001**

When a user submits a valid email address to access the platform, the system shall send a magic link to that email address.

**EARS-AUTH-002**

When the system generates a magic link for a user, the system shall generate a unique, single-use link with a maximum validity of 15 minutes.

**EARS-AUTH-003**

When the system generates a new magic link for the same user, the system shall invalidate any previously generated magic link for that user.

**EARS-AUTH-004**

When a user submits an email address that is not yet associated with an account, the system shall create the user account before sending the magic link.

**EARS-AUTH-005**

If a magic link request cannot be processed, then the system shall return a generic authentication message without revealing whether the email exists.

---

## 1.2 Uso y validación del magic link

**EARS-AUTH-006**

When a user opens a valid magic link, the system shall authenticate the user and start a session.

**EARS-AUTH-007**

When a valid magic link is used successfully, the system shall invalidate that magic link immediately after use.

**EARS-AUTH-008**

If a user opens an expired magic link, then the system shall reject the authentication attempt.

**EARS-AUTH-009**

If a user opens a magic link that has already been used, then the system shall reject the authentication attempt.

**EARS-AUTH-010**

If a magic link authentication attempt is rejected, then the system shall display a clear error message to the user.

---

# 2. Comunidades privadas

## 2.1 Creación de comunidad

**EARS-COM-001**

When an authenticated user creates a community, the system shall create it as a private community.

**EARS-COM-002**

When an authenticated user creates a community, the system shall require a name with a length between 1 and 100 characters.

**EARS-COM-003**

When an authenticated user creates a community, the system shall require a description with a length between 1 and 300 characters.

**EARS-COM-004**

When an authenticated user creates a community, the system shall require a community image.

**EARS-COM-005**

If the uploaded community image is missing or is not a valid image format, then the system shall reject the community creation request.

**EARS-COM-006**

When a community is created successfully, the system shall assign the creator as community admin.

---

## 2.2 Acceso a comunidad

**EARS-COM-007**

If a user is not a member of a community, then the system shall deny access to that community.

**EARS-COM-008**

When a community member accesses a community, the system shall allow access only to the community to which that member belongs.

---

## 2.3 Invitaciones a comunidad

**EARS-COM-009**

When a community admin generates an invitation link, the system shall generate a unique invitation link associated with exactly one community.

**EARS-COM-010**

When the system generates an invitation link, the system shall generate it as non-predictable.

**EARS-COM-011**

When a community admin generates an invitation link, the system shall allow generating multiple invitation links for the same community.

**EARS-COM-012**

When a user opens a valid invitation link for a community, the system shall add the user as a member of that community.

**EARS-COM-013**

When a user is added to a community through an invitation link, the system shall register the user identifier, the community identifier, and the join timestamp.

**EARS-COM-014**

If an invitation link is expired, then the system shall reject its use.

**EARS-COM-015**

If an invitation link has been revoked, then the system shall reject its use.

**EARS-COM-016**

If an invitation link has already been used, then the system shall reject its use.

**EARS-COM-017**

If a user who is already a member uses a valid invitation link for the same community, then the system shall ignore the request and shall not create a duplicate membership.

**EARS-COM-018**

When a user joins a community through an invitation link, the system shall consume that link so it cannot be used again.

---

# 3. Proyectos

## 3.1 Contexto de comunidad

**EARS-PROJ-001**

When a user creates a project, the system shall assign that project to exactly one community.

**EARS-PROJ-002**

If a user is not a member of the community to which a project belongs, then the system shall deny access to that project.

**EARS-PROJ-003**

When a community member accesses a project, the system shall allow viewing that project only if the member belongs to the same community as the project.

**EARS-PROJ-004**

When a community member interacts with a project, the system shall limit interaction to viewing the project and submitting feedback, according to the project state rules.

**EARS-PROJ-005**

When a community member creates a project inside a community, the system shall allow the creation.

---

## 3.2 Galería de imágenes

**EARS-PROJ-006**

When a user creates a project, the system shall require at least one image in the project gallery.

**EARS-PROJ-007**

When a user creates or updates a project, the system shall allow a maximum of five images in the project gallery.

**EARS-PROJ-008**

If a user attempts to save a project with zero images, then the system shall reject the operation.

**EARS-PROJ-009**

If a user attempts to save a project with more than five images, then the system shall reject the operation.

**EARS-PROJ-010**

When a user updates a project, the system shall continue enforcing the minimum of one image and the maximum of five images.

---

# 4. Estados del proyecto

## 4.1 Estados funcionales de plataforma

**EARS-STATE-001**

When a project is in Draft state, the system shall make it visible only to its creator.

**EARS-STATE-002**

When a project is in Live state, the system shall make it visible to all members of the same community.

**EARS-STATE-003**

When a project is in Inactive state, the system shall make it visible to all members of the same community.

---

## 4.2 Feedback por estado

**EARS-STATE-004**

When a project is in Live state, the system shall allow community members to submit feedback on that project.

**EARS-STATE-005**

If a project is in Draft state, then the system shall not allow feedback submission.

**EARS-STATE-006**

If a project is in Inactive state, then the system shall not allow feedback submission.

---

## 4.3 Recomendaciones derivadas de feedback

**EARS-STATE-007**

When the system presents a project recommendation based on community feedback, the system shall present one of the following recommendation values: Seguir iterando, Impulsar, or Pausar.

**EARS-STATE-008**

When the system presents a recommendation based on community feedback, the system shall not change the project platform state as a result of that recommendation.

---

# 5. Perfil de usuario

## 5.1 Existencia y completado

**EARS-PROFILE-001**

When a user account exists, the system shall provide a user profile for that user.

**EARS-PROFILE-002**

When a user chooses to complete the profile, the system shall allow editing the optional profile information.

---

## 5.2 Contenido del perfil

**EARS-PROFILE-003**

When a user edits the profile, the system shall allow updating the user interests.

**EARS-PROFILE-004**

When a user edits the profile, the system shall allow updating the short bio.

**EARS-PROFILE-005**

When the system displays a user profile, the system shall include the user interests when available.

**EARS-PROFILE-006**

When the system displays a user profile, the system shall include the short bio when available.

**EARS-PROFILE-007**

When the system displays a user profile, the system shall include the number of feedback items submitted by the user.

**EARS-PROFILE-008**

When the system displays a user profile, the system shall include the number of projects created by the user.

**EARS-PROFILE-009**

When a user updates profile interests, the system shall store those interests for future matching functionality.

---

## 5.3 Visibilidad del perfil

**EARS-PROFILE-010**

When a user views another user profile, the system shall allow the profile to be shown only if both users belong to the same community.

**EARS-PROFILE-011**

If two users do not share a community, then the system shall deny access to the profile view between them.

**EARS-PROFILE-012**

When the system shows a user profile inside a community context, the system shall show the same global profile for that user.

---

# 6. Interacciones entre módulos

**EARS-INT-001**

If a user does not belong to the community of a project, then the system shall deny access to that project.

**EARS-INT-002**

When a user submits feedback on a project, the system shall associate that feedback with the submitting user profile.

**EARS-INT-003**

When the system stores feedback, the system shall associate it with exactly one user and exactly one project.

**EARS-INT-004**

If a project is not in Live state, then the system shall not allow feedback submission even if the user belongs to the community.

---

# 7. Restricciones no funcionales formalizables para MVP

**EARS-NFR-001**

When the system generates a magic link, the system shall generate it in a non-predictable manner.

**EARS-NFR-002**

When the system generates an invitation link, the system shall generate it in a non-predictable manner.

**EARS-NFR-003**

If a user attempts to access a private community by direct URL without authorization, then the system shall deny access.

**EARS-NFR-004**

If a user attempts to access a project or profile outside the allowed community scope, then the system shall deny access.

**EARS-NFR-005**

When the system processes community-scoped data, the system shall limit visibility of that data to authorized members of the corresponding community.

**EARS-NFR-006**

When the system processes a magic link request, the system shall target delivery in less than 30 seconds under normal operating conditions.

---

# 8. Requisitos descartados o fuera de alcance en este MVP

Estos no se formalizan como comportamiento contractual del MVP:

- reactivación de proyectos desde Inactive,
- eliminación de usuarios de comunidad,
- reingreso de usuarios eliminados,
- invalidación de sesión por expulsión de comunidad,
- lógica operacional de las recomendaciones más allá de su presentación.

---

# 9. Observaciones de calidad del contrato EARS

El contrato queda ahora **cerrado y trazable** para MVP porque:

- los estados del proyecto ya no están mezclados con recomendaciones,
- las invitaciones tienen semántica cerrada,
- el perfil tiene modelo global y regla de visibilidad definida,
- feedback tiene restricciones de estado y asociación claras,
- se ha excluido explícitamente lo que no entra en MVP.
